01 — The Problem

GitHub gives you raw activity.
Not intelligence.

Your team merges pull requests every day. GitHub shows you who opened what. But it never tells you: "this PR touches the authentication system" or "this one changed 800 lines across 20 files — it's high risk." You find out about risky merges after things break in production.

🔍
No risk visibility
You can't tell from the PR title whether it's a 2-line typo fix or a 500-line overhaul of your core API. Both look the same in GitHub's PR list.
⏱️
No review analytics
Which reviewer responds fastest? Which PRs sit waiting for days? Which engineer carries the most review load? GitHub doesn't answer any of this.
📉
No velocity tracking
Are PRs getting bigger over time? Is merge speed slowing down? Is your team's output improving or declining? You only know if you build something to measure it.
🐢
Delayed reporting
Most tools give you a weekly digest or a dashboard that refreshes every hour. By the time you see a high-risk PR, it may already be merged.

GitHub alone
  • ✗No automated risk scoring

  • ✗No reviewer analytics

03 — Architecture

The pipeline, step by step

PRPulse uses an event-driven architecture. No service talks to another service directly. They communicate through Redis Streams — which means if one crashes, the others keep running and no data is lost.

01
GitHub
Developer opens a Pull Request
A developer pushes code and opens a PR on GitHub. GitHub automatically fires a webhook — a POST request — to any URL you tell it to. PRPulse registers its webhook URL when you install the GitHub App.
pull_request.opened JSON payload HMAC signature
02
Webhook Service — FastAPI :8000
Receive, validate, and publish to Redis
The webhook service receives the POST from GitHub. It extracts the PR number from the JSON body, then calls XADD prpulse:events:raw to publish the event into the Redis Stream. It returns HTTP 200 in under 50ms. That's it — it does nothing else. No database. No analysis. Just write to Redis and respond fast.
XADD prpulse:events:raw HTTP 200 in <50ms
03
Redis Streams
Event waits in the queue
The event now sits in prpulse:events:raw, a Redis Stream. Think of it like a logbook — entries are written in order and stay there until processed. Even if the worker is busy or crashed, this event will not disappear. Redis tracks which messages have been "picked up but not confirmed" in a Pending Entries List (PEL). If the worker crashes mid-processing, the event is reclaimed automatically after 60 seconds.
Pending Entries List Consumer Group Never lost
04
Analysis Worker — Python async loop
Pick up the event, fetch PR data, analyze
The worker runs an infinite loop calling XREADGROUP. When it sees a new message, it picks it up. Then it calls the GitHub API to get the full PR — metadata (author, title) and all changed files with their diffs. It runs those through the diff parser and risk scorer. If GitHub's API fails, it retries up to 3 times with increasing delays (1s → 2s → 4s). If all retries fail, the event goes to the Dead Letter Queue.
XREADGROUP GitHub API Pagination (100 files/page) 3 retries + backoff DLQ on failure
05
Diff Parser + Risk Scorer
What changed? How risky is it?
The diff parser looks at every file's patch text. It counts lines added and removed. It detects four types of signals: a config file was changed, a large chunk of code was rewritten, a function signature was modified, or an API route was deleted. Then the risk scorer adds points for each signal found and classifies the PR as LOW, MEDIUM, or HIGH risk.
config file changed → +15 points lines_added > 300 → +20 points lines_removed > 200 → +10 points any breaking change found → +40 points files_changed > 10 → +15 points ──────────────────────────────────── 0–30 LOW │ 31–60 MEDIUM │ 61–100 HIGH
4 signal types Additive scoring Deduplication
06
PostgreSQL — Persistence
Write results to database, then acknowledge
The worker writes two rows to PostgreSQL: one in pull_requests (PR number, author, repo) and one in pr_analysis (lines changed, risk score, risk level). The database enforces a foreign key — pr_analysis.pr_number must exist in pull_requests first. After both rows are committed, the worker calls XACK to tell Redis the message is done. XACK only fires after the DB write succeeds. This ordering guarantees nothing is lost even if the worker crashes between writing and acknowledging.
ON CONFLICT DO NOTHING Foreign key enforced XACK after DB write asyncpg pool
07
Query API + Dashboard — Coming in Phase 6
Read data, stream live updates
The Query API (FastAPI :8001) reads from PostgreSQL and exposes REST endpoints for PR lists, risk distributions, and metrics. After the worker persists a result, it also publishes to prpulse:notifications. A broadcaster reads that stream and pushes events to all connected dashboard clients via Server-Sent Events. The React dashboard shows a live PR feed — new PRs appear automatically when analyzed, no refresh needed.
REST API SSE /events/stream React + EventSource Live PR feed
04 — Database Schema

What we store in PostgreSQL

Two tables, one relationship. Simple and deliberate. The foreign key between them is the guarantee that analysis data always has a parent PR.

🗂️ pull_requests
idPK · SERIAL
pr_numberINT · UNIQUE
authorTEXT
repo_ownerTEXT
repo_nameTEXT
files_changedINT
created_atTIMESTAMP
📊 pr_analysis
idPK · SERIAL
pr_numberFK → pull_requests
files_changedINT
lines_addedINT
lines_removedINT
risk_scoreFLOAT
risk_levelTEXT — LOW/MEDIUM/HIGH
analyzed_atTIMESTAMP
Why two tables? A PR is a fact — it was opened at a moment in time. Analysis is computed later, potentially re-computed if the formula changes. Separating them means we can update risk scoring logic and reprocess PRs without touching the original PR data. The UNIQUE on pr_number combined with ON CONFLICT DO NOTHING makes the insert idempotent — safe to retry as many times as needed.

Q & A — Phase 1
QWhat is a webhook and how does GitHub use it?
A webhook is just an HTTP POST that GitHub sends to your URL whenever something happens in your repo — like someone opening a PR. It's GitHub's way of saying "hey, something just happened, here's the data." Your server receives it, does something with it, and returns HTTP 200 to say "got it." If you return an error or take too long, GitHub thinks delivery failed and retries.
QWhy not just process the PR directly inside the webhook handler?
Processing a PR means calling GitHub's API twice (metadata + files), parsing diffs, running scoring, and writing to the database. That could take 5–10 seconds. GitHub expects a response in 1–2 seconds max. If you exceed that, GitHub marks the delivery as failed and starts retrying. By writing to Redis and returning 200 immediately, the webhook handler is always fast. The actual analysis happens in the background.
QWhat is Redis Streams and how is it different from a normal Redis list?
A Redis List is like a queue — once you pop an item, it's gone. Redis Streams is like a logbook — entries stay even after they're read. This matters for reliability: if the worker crashes while processing message #5, that message stays in the Pending Entries List and gets retried. With a plain list, it would be gone forever. Streams also have Consumer Groups, which let multiple workers share the load without processing the same message twice.

  • Reprocessed through the same pipeline
Q & A — Phase 2
QWhat is the Pending Entries List (PEL)?
When XREADGROUP delivers a message to a worker, Redis moves that message to the PEL — a list of "in-flight" messages. The message stays there until the worker calls XACK to confirm it was processed. If the worker crashes before calling XACK, the message is stuck in the PEL. Our recovery loop calls XAUTOCLAIM every 30 seconds to find messages idle in the PEL for over 60 seconds and reclaims them for reprocessing. Nothing is ever permanently lost.
QWhy does XACK fire AFTER the database write and not before?
If you call XACK before writing to the database, and then the database write fails, you've told Redis "done" but actually nothing was saved. The message won't be retried because Redis thinks it succeeded. By calling XACK only after a successful DB write, we guarantee that every acknowledged message is also saved. If the DB write fails, XACK is never called, the message stays in the PEL, and the recovery loop retries it.
QWhat happens after all 3 retries fail?
The message is written to a separate Redis stream called prpulse:events:failed — the Dead Letter Queue. This stream stores the error message, how many retries were attempted, and the original PR number. The original message is then XACK'd so it doesn't block other messages. A developer can later read the DLQ, fix the root cause, and replay those events manually.

Q & A — Phase 3
QHow does the diff parser detect a function signature change?
GitHub gives us the raw "patch" text for each modified file. Lines starting with "-" were removed, lines starting with "+" were added. The parser runs a regex looking for removed lines containing "def function_name(" (Python) or "function name(" (JavaScript). If it finds a removed definition AND an added definition in the same patch, it flags this as a function signature change — meaning the function parameters may have changed, breaking callers.
QWhy does the GitHub client reuse one httpx.AsyncClient?
Creating a new HTTP client every time means creating a new TCP connection to GitHub's servers for every API call — which includes a "handshake" that adds 100–300ms latency. A shared client keeps the TCP connection alive (connection pooling) and reuses it. For a worker that may process hundreds of PRs, this is significantly faster. It's created once at startup and reused for the entire process lifetime.

Q & A — Phase 4
QWhat does idempotent mean and why does it matter for inserts?
Idempotent means "you can run the same operation 10 times and get the same result as running it once." Our insert is idempotent because of ON CONFLICT DO NOTHING. If the worker retries a PR it already processed, the second INSERT just silently skips without error. Without this, the DB would throw a duplicate key error and the message would keep retrying forever, always failing at the insert step. Idempotency makes retry logic safe.
QWhy is malformed input (like pr_number = "abc") checked before the retry loop?
If "abc" reaches the retry loop, each attempt will fail with ValueError: invalid literal for int() — a permanent error that retrying will never fix. Without the early check, all 3 retry slots get burned on an unrecoverable input error with 3 useless backoff waits. By catching it before the loop, we discard it cleanly with one XACK and move on immediately. This is the difference between a smart system and a system that wastes time on bad data.

Q & A — Phase 5
QWhy is the API split into Router, Service, and Repository layers?
Each layer has exactly one job. The Router handles HTTP (receives request, returns response) — no SQL, no business logic. The Service applies business rules (which repo to look up, how to format results) — no HTTP, no SQL. The Repository writes raw SQL and talks to the database — nothing else. If you switch from PostgreSQL to a different database, you only change the Repository layer. The Router and Service don't care what database is behind it. This separation makes the code easier to test and change.
QWhat is an asyncpg connection pool and why is it better than one connection?
A connection pool is a group of database connections that are created once and reused. Creating a new database connection for every HTTP request is slow (takes ~50ms). With a pool of 10 connections, up to 10 requests can query the database simultaneously without waiting. When a request finishes, its connection goes back to the pool for the next request. This is how production APIs handle hundreds of concurrent users without slowing down.

Q & A — Phase 6
QWhat are Server-Sent Events (SSE) and how do they work?
SSE is a way for a server to push data to a browser over a regular HTTP connection that never closes. The browser calls a URL (like /events/stream) and gets a response with Content-Type: text/event-stream. The server keeps this connection open and sends chunks of data formatted as data: {...}\n\n. Every time the server sends a chunk, the browser fires a JavaScript event. It's simpler than WebSockets because it's just one-way (server → browser) and built on plain HTTP.
QWhen the dashboard loads, how does it show existing PRs that were analyzed before the page opened?
SSE only delivers future events — it doesn't replay history. So the dashboard does two things on load: first, it calls GET /repos/{id}/prs via a normal REST API call to load existing PRs from the database. Second, it opens an SSE connection to /events/stream for new ones going forward. Both together give the complete picture: historical data from the REST call, live updates from SSE.

From "PR opened" to

Here is exactly what happens at the system level, step by step, from the moment a developer opens a pull request.

1
Developer opens a PR on GitHub
Code is pushed to a branch, a pull request is created via the GitHub UI or git push.
GitHub UI / git CLI
2
GitHub fires a webhook POST to PRPulse
GitHub sends a POST request to the registered webhook URL with the PR payload (JSON containing pr_number, author, repo, etc.) and an HMAC-SHA256 signature header for security.
POST /webhook · X-GitHub-Event: pull_request
3
Webhook service extracts pr_number → writes to Redis
FastAPI handler reads the JSON body, gets pr_number. Calls XADD prpulse:events:raw {"event_type": "pr.opened", "pr_number": "42"}. Redis assigns it a unique stream ID based on the current timestamp.
XADD prpulse:events:raw · ~2ms
4
Webhook service returns HTTP 200 to GitHub
Total time for steps 2→3→4 is under 50ms. GitHub receives 200, marks delivery as successful, and closes the connection. GitHub never needs to retry this delivery.
HTTP 200 · <50ms total
5
Worker picks up the event from Redis Streams
The worker's XREADGROUP loop (blocking, checks every 5 seconds) sees the new message and picks it up. The message moves to the Pending Entries List — it is now "in progress." If the worker crashes here, the recovery loop reclaims it after 60 seconds.
XREADGROUP · message → PEL
6
Worker calls GitHub API: fetch PR metadata
GET /repos/{owner}/{repo}/pulls/42 — returns the full PR object: title, author, created_at, base branch, head branch, and the diff URLs. The single reused httpx.AsyncClient sends this over an existing TCP connection.
GitHub REST API · GET /pulls/42
7
Worker calls GitHub API: fetch all changed files (with pagination)
GET /repos/{owner}/{repo}/pulls/42/files?per_page=100 — returns each file with its patch text (the diff). If the PR touches more than 100 files, the worker fetches page 2, page 3, etc., until all files are collected.
GET /pulls/42/files · pagination · patch text
8
Diff parser analyzes each file
For every file: count lines added ("+") and removed ("-"). Check filename patterns for config file changes. Check line thresholds for large file changes. Run regex on patch text for function signature changes and route deletions. Deduplicate by (signal_type, filename) pairs.
4 signal types · deduplication
9
Risk scorer assigns a score and level
Add up points: +15 for many files, +20 for large additions, +10 for large deletions, +40 if any breaking change was found, +15 if a config file changed. Clamp result to 0–100. Classify: 0–30 = LOW, 31–60 = MEDIUM, 61–100 = HIGH.
Additive rules · LOW / MEDIUM / HIGH
10
Worker writes to PostgreSQL (two rows)
First: INSERT INTO pull_requests (pr_number, author, repo_owner, repo_name) ON CONFLICT DO NOTHING. Then: INSERT INTO pr_analysis (pr_number, lines_added, lines_removed, risk_score, risk_level). Both writes happen in this order — FK constraint on pr_analysis requires pull_requests to exist first.
INSERT ON CONFLICT DO NOTHING · FK enforced
11
Worker calls XACK — message is done
Only now, after both DB writes commit successfully, does the worker call XACK prpulse:events:raw prpulse-analysis-workers {message_id}. This removes the message from the Pending Entries List permanently. If DB writes failed, XACK is never called and the message will be retried.
